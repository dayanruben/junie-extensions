# Laravel Engineer

## Core Workflow

1. **Check toolchain** — verify Laravel Boost, Larastan, Pint, ide-helper are installed (see Setup Check)
2. **Read project guidelines** — load in order: `.ai/guidelines/` (Boost), `.junie/guidelines.md`, `AGENTS.md` — see Setup Check
3. **Design first** — for non-trivial tasks, confirm approach before coding
4. **Implement** — follow Laravel conventions: thin controllers, Form Requests for validation, Services/Actions for business logic
5. **Post-generation pipeline** — run Pint → Larastan → simplifier pass before presenting code
6. **Test** — write Pest tests; run `./vendor/bin/pest` to confirm green (or `./vendor/bin/phpunit` if project uses PHPUnit)

## Setup Check

On first use in a Laravel project, verify the toolchain:

```
composer.json exists?
├── YES → check installed tools:
│   ├── laravel/boost missing        → composer require --dev laravel/boost
│   │                                  then: php artisan boost:install
│   ├── vendor/bin/phpstan missing   → composer require --dev larastan/larastan
│   ├── vendor/bin/pint missing      → composer require --dev laravel/pint
│   ├── barryvdh/laravel-ide-helper missing → composer require --dev barryvdh/laravel-ide-helper
│   │                                          then: php artisan ide-helper:generate
│   │                                                php artisan ide-helper:models --nowrite
│   └── vendor/bin/rector missing    → composer require --dev rector/rector
└── NO → not a Laravel project, skip
```

After verifying, load project-specific guidelines in this order:
1. `.ai/guidelines/` exists (generated by Boost) → load version-specific rules (Livewire, Filament, Tailwind versions)
2. `.junie/guidelines.md` exists → load project architecture, anti-patterns, service patterns
3. `AGENTS.md` exists → load agent-specific conventions
- `phpstan.neon` missing → create minimal config pointing to `larastan/larastan` extension at level 5

## Quick Start

### Model

```php
namespace App\Models;

use Illuminate\Database\Eloquent\Builder;
use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsTo;
use Illuminate\Database\Eloquent\Relations\HasMany;
use Illuminate\Database\Eloquent\SoftDeletes;

class Order extends Model
{
    use HasFactory, SoftDeletes;

    protected $fillable = ['user_id', 'status', 'total'];

    protected $casts = [
        'total'      => 'decimal:2',
        'status'     => OrderStatus::class,  // enum cast
        'metadata'   => 'array',
        'shipped_at' => 'datetime',
    ];

    public function user(): BelongsTo
    {
        return $this->belongsTo(User::class);
    }

    public function items(): HasMany
    {
        return $this->hasMany(OrderItem::class);
    }

    // Scope
    public function scopePending(Builder $query): void
    {
        $query->where('status', OrderStatus::Pending);
    }
}
```

### Controller

```php
namespace App\Http\Controllers\Api;

use App\Http\Controllers\Controller;
use App\Http\Requests\StoreOrderRequest;
use App\Http\Resources\OrderResource;
use App\Services\OrderService;
use Illuminate\Http\Resources\Json\AnonymousResourceCollection;

class OrderController extends Controller
{
    public function __construct(
        private readonly OrderService $orderService,
    ) {}

    public function index(): AnonymousResourceCollection
    {
        $orders = Order::with('items')->paginate(20);
        return OrderResource::collection($orders);
    }

    public function store(StoreOrderRequest $request): OrderResource
    {
        $order = $this->orderService->create($request->validated());
        return new OrderResource($order);
    }
}
```

### Form Request

```php
namespace App\Http\Requests;

use Illuminate\Foundation\Http\FormRequest;

class StoreOrderRequest extends FormRequest
{
    public function authorize(): bool
    {
        return true;
    }

    public function rules(): array
    {
        return [
            'items'              => ['required', 'array', 'min:1'],
            'items.*.product_id' => ['required', 'integer', 'exists:products,id'],
            'items.*.quantity'   => ['required', 'integer', 'min:1'],
        ];
    }
}
```

### API Resource

```php
namespace App\Http\Resources;

use Illuminate\Http\Request;
use Illuminate\Http\Resources\Json\JsonResource;

class OrderResource extends JsonResource
{
    public function toArray(Request $request): array
    {
        return [
            'id'         => $this->id,
            'status'     => $this->status,
            'total'      => $this->total,
            'items'      => OrderItemResource::collection($this->whenLoaded('items')),
            'created_at' => $this->created_at,
        ];
    }
}
```

## Post-Generation Pipeline

After generating any code, always run in this order:

1. **Format** — `./vendor/bin/pint` — auto-fixes style to PSR-12 + Laravel conventions
2. **Analyse** — `./vendor/bin/phpstan analyse` — catches type errors and Laravel anti-patterns
3. **Simplifier pass** — review generated code for:
   - Unnecessary nesting → flatten with early returns
   - Verbose conditionals → match expressions or ternary
   - Raw queries where Eloquent suffices
   - Missing `whenLoaded()` in Resources
   - Inline logic that belongs in a scope or service

## Running Tests

Always run tests after implementing a feature. Use whichever runner the project has installed:

```bash
# Pest (preferred for Laravel projects)
./vendor/bin/pest

# Run a specific test file
./vendor/bin/pest tests/Feature/OrderTest.php

# Run with coverage (requires Xdebug or PCOV)
./vendor/bin/pest --coverage --min=80

# PHPUnit (if project uses PHPUnit instead of Pest)
./vendor/bin/phpunit

# Via artisan (auto-detects Pest or PHPUnit)
php artisan test
php artisan test --filter=OrderTest
php artisan test --parallel
```

Auto-detect which runner is installed:
- `composer.json` has `pestphp/pest` → use `./vendor/bin/pest`
- `composer.json` has `phpunit/phpunit` only → use `./vendor/bin/phpunit`

## Large Feature Planning

Before implementing a large feature (new module, refactor, multi-file change):
- Suggest running `npx repomix` to pack the codebase into a single context
- Confirm architecture approach before writing code
- Identify which models, services, and migrations are affected

## Constraints

### MUST DO

| Rule | Correct Pattern |
|------|----------------|
| Validate in Form Requests | `class StorePostRequest extends FormRequest` |
| Transform in Resources | `class PostResource extends JsonResource` |
| Business logic in Services or Actions | `OrderService::create()`, `CreateOrderAction::handle()` |
| Use `$fillable` or `$guarded` | Never leave both empty |
| Eager-load relationships | `Order::with('items', 'user')->get()` |
| Use casts for types | `'status' => OrderStatus::class` in `$casts` |
| Type-hint injected services | `private readonly OrderService $service` |

### MUST NOT DO

- Put business logic in controllers
- Use `DB::table()` when an Eloquent model exists
- Call `->get()` inside a loop (N+1)
- Use `$request->all()` for mass assignment
- Leave relationships un-eager-loaded in collection endpoints
- Return raw arrays from controllers instead of Resources
- Use `Auth::user()` directly in service layer — pass user as parameter
- Use `app()` or `resolve()` as service locator — use constructor injection instead
- Store passwords in plain text — use `Hash::make()`, never raw `md5()` or `sha1()`

## Output Format

When implementing a feature, deliver in this order:
1. Migration (if schema changes needed)
2. Model with casts, relationships, scopes
3. Form Request + Service / Action
4. Controller + API Resource
5. Pest tests
6. One-line explanation of key architecture decisions
